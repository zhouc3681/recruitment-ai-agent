# HR 招聘数据自动录入助手

React 版招聘自动化 Agent PoC，重点演示“输入/监听 -> 抽取 -> 核验 -> 入库 -> 看板同步”的主流程。它不是复杂 ATS，而是一个低成本招聘数字员工雏形，用来替代 HR 在企业微信、微信群、腾讯文档之间反复复制、记录和跟进的工作。

## 运行方式

## 页面结构

- 招聘驾驶舱：展示今天新增、待确认、待补充、已入库、今日同步、待核验、面试通过率、录入提效、渠道分布、每日新增和招聘漏斗。
- 效率提升测算：展示人工录入 vs AI 录入、每日/每周/每月节省时间、预计减少人工录入比例。
- Agent 工作流：支持企业微信群记录、企业微信私聊、腾讯文档表格、面试反馈文本输入模板，并展示消息气泡/文档导入预览。
- 候选人中心：候选人搜索、岗位/来源/阶段/完整度筛选、排序、左右布局详情、审计链、同步状态、底部原始记录与 JSON 预览。
- 知识库中心：展示 JD Match Skill、岗位 JD 库、招聘标准库、历史录用/淘汰案例；点击不同 JD 会联动更新 Agent 可解释回答。

## 已实现能力

- 消息采集 Agent：用 Webhook 模拟企业微信群、企业微信私聊、腾讯文档监听。
- 顶部六步流程：消息进入 -> AI抽取 -> 缺失核验 -> HR确认 -> 自动入库 -> 看板同步，打开页面即可理解主流程。
- 一键演示模式：15-20 秒内自动播放“收到消息、AI处理中、字段抽取完成、生成回访草稿、HR确认、写入数据库、看板刷新”。
- 来源样式预览：企业微信场景展示消息气泡，腾讯文档场景展示表格导入样式。
- 批量粘贴抽取：支持一次粘贴多名候选人，腾讯文档表格文本会按行解析。
- 字段高亮与来源引用：每个字段显示来源文本，便于 HR 快速复核。
- 字段审计链：记录字段名称、字段值、来源渠道、定位行号和原文引用。
- 审计链修正：面试安排仅匹配真实面试时间、约面、面试阶段或面试官信息，不再把“面试台账/面试安排经历”误判为面试安排。
- 缺失核验：完整度低于 80% 生成 AI 回访草稿。
- Bad Case 处理：置信度 95% 以上自动入库，80%-95% 待 HR 确认，80% 以下进入待核验池。
- 实时事件流：展示收到消息、字段抽取、缺失核验、回访草稿、写入数据库、看板同步。
- 批量入库：HR 勾选多名候选人后统一确认写入本地候选人库。
- 状态机：根据面试安排、反馈、结果自动推进阶段。
- JSON 预览：支持复制、下载、入库前检查。
- 看板同步：候选人入库后驾驶舱和候选人中心实时更新。
- 单文件演示：`demo-preview.html` 可直接双击打开，无需命令行。
- 候选人详情去代码化：Explainability 区域展示推荐原因、审计链和同步摘要；代码式 JSON 仅保留在 Agent 工作流的标准化 JSON 预览中。
- RAG 联动解释：知识库中心点击任一 JD，右侧解释会同步切换匹配评分、使用的 JD 规则、风险原因和历史案例引用。

## 低成本生产化路径

PoC 当前用 React + localStorage 模拟数据库。正式接入时建议保持以下替换边界：

- 前端：React 保持不变。
- 数据库：localStorage 替换为 SQLite，后续可升级 PostgreSQL / ATS。
- 向量库：内置 RAG mock 替换为 Chroma。
- 消息接入：Webhook 模拟替换为企业微信应用回调、群机器人、腾讯文档 API。
- LLM：本地规则抽取替换为 OpenAI 兼容接口，使用 JSON Schema 约束输出。
- HITL：当前按钮确认替换为企业微信审批卡片或内部工作台审批。

## 状态机

```text
简历筛选 -> 待沟通 -> 待面试 -> 一面 -> 复试 -> Offer -> 入职
                                           -> 淘汰
```

Agent 会根据消息关键词和结构化字段自动推进：

- 出现联系方式：进入待沟通。
- 出现面试安排/约面：进入待面试。
- 出现一面/面试反馈/待定：进入一面。
- 出现复试/二面：进入复试。
- 出现 Offer：进入 Offer。
- 出现淘汰/不通过：进入淘汰。
